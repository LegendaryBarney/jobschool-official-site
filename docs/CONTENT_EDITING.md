- **改 DB + Redeploy** = 在 Supabase 後臺直接改資料 → **多做一步「重新部署」**（見 §1.4）才會反映到官網。

## 1. 資料放在哪裡、改了怎麼生效

### 1.1 網站的資料只放在兩種地方

1. **repo 裡的檔案**（`src/content/` 下的 JSON 與 Markdown）：機構事實、價格政策、課程/師資/文章等「內容」。
2. **Supabase 資料庫**（線上後臺，網址與帳號由工程師提供）：課表時段、課程清單（給課表與試聽表用）、試聽報名紀錄、程式設計課的即時價格。

除此之外**沒有第三個地方**。任何頁面上看到的文字，都能用 §0 的表回推到這兩處之一。

### 1.2 防呆機制：改壞了不會悄悄上線

`info.json` 與 `policy.json` 都掛了格式檢查（工程上叫 schema 驗證）。如果改的時候少打一個引號、刪錯一個欄位，網站**建置時會直接報錯、不會部署**——等於改壞了會被擋下來，不會有「悄悄變成壞頁面」的風險。所以放心改，錯了會知道。

### 1.3 檔案類的上線流程（現行協定）

1. 改檔案（自己改或請工程師改）
2. 存成一次 commit，推到 feature branch（**不直接動 main**）
3. Vercel 自動產生一個 **preview 網址**——先在這裡確認改對了
4. 確認無誤 → 由業主拍板合併進 main → **正式上線自動部署**

### 1.4 資料庫類的生效方式：改完要「重新部署（Redeploy）」

官網頁面是**事先做好的靜態頁**（這是它快又穩的原因）。頁面在「部署的那一刻」讀一次資料庫，把結果印在頁面上。所以：

> **在 Supabase 改了資料，官網不會立刻變。要觸發一次重新部署，讓網站重新讀一次資料庫。**

怎麼 Redeploy（兩選一）：

- **Vercel 後臺（建議）**：登入 vercel.com → 專案 `jobs_official_site` → 「Deployments」→ 找到最新的 Production 部署 → 右側「⋯」→ **Redeploy**。約 2-3 分鐘完成。
- **請工程師**推任意一個 commit 到 main（任何上線都會順帶重讀資料庫）。

> 小知識：試聽表單送出後的「後端檢查」是即時查資料庫的，但**表單畫面上的選項**是部署時做好的。所以改完課表若不 Redeploy，家長看到的選項會是舊的——改 DB 後請一定記得這一步。

## 2. 機構事實：`src/content/site/info.json`

全站「這間補習班是誰、在哪、怎麼聯絡」的**唯一權威來源**。改這個檔，Footer、聯絡頁、關於頁、首頁統計、社群分享預覽圖（OG 圖）、Google 結構化資料（JSON-LD）、llms.txt 全部自動跟著變，**不需要再去各頁面改字**。

### 2.1 逐欄位說明

| 欄位 | 是什麼 | 目前的值（範例） |
|---|---|---|
| `name` / `legalName` | 對外名稱 / 正式名稱 | 賈伯斯數理教室 |
| `url` | 官網網址 | `https://jobsedu.com.tw` |
| `descriptionTemplate` | 品牌一句話簡介。裡面的 `{years}` 是佔位符，會自動代入「今年 − 創立年」 | 「…2014 年返鄉創立、品牌資歷 {years} 年…」 |
| `founded` | 創立年份。**年資（brandYears）與 Footer 版權年份（2014–今年）都由它自動推算，每年不用手改** | `2014` |
| `phone.e164` | 電話的國際格式（給「點了直接撥號」用） | `+886-5-223-0303` |
| `phone.display` | 電話的顯示格式（頁面上看到的樣子） | `(05) 223-0303` |
| `email` | 聯絡信箱 | `contact@jobsedu.com.tw` |
| `hours.openingHours` | 營業時間（給 Google 的簡寫格式） | `["Mo-Fr 17:00-21:30"]` |
| `hours.display` | 營業時間顯示文字 | `週一至週五 17:00–21:30（週六日休）` |
| `hours.opens` / `closes` / `days` | 開門 / 關門時間與營業日（給 Google 結構化資料） | `17:00` / `21:30` / Monday…Friday |
| `socials` | 四個社群連結預設值：`instagram` / `facebook` / `youtube` / `googleBusiness` | （見檔案） |
| `serviceArea` | 服務區域（Google 結構化資料用） | `嘉義市` |
| `locations.jobs` | 賈伯斯本部：結構化地址（街道 / 行政區 / 郵遞區號）、完整地址字串 `full`、電話、地圖座標 `geo` | 嘉義市東區康樂街 10 號 |
| `locations.shinobi` | 忍文理教室（姊妹品牌）：同上，另有 `foundedYear`（2023）與 `website`（官網連結，全站可點的「忍文理」都指到這裡） | 嘉義市西區杭州一街 38 號 |
| `stats.studentsServed` / `studentsServedDisplay` | 累積學生數（數字 / 顯示文字） | `1000` / `1000+` |
| `stats.classSizeMin` / `classSizeMax` / `classTiers` | 班級規模上下限與三級距文字 | 極小班 2-5 人 / 精緻班 6-10 人 / 小班 10-14 人 |
| `stats.maxTeacherYears` | 最資深老師年資 | `15` |
| `responseSla` | 「我們會在多久內回覆」的對外承諾，全站十多處統一引用 | `1 個工作日` |
| `nearbySchoolsNote` | 教室鄰近學校的一句話（聯絡頁用） | `距嘉中、嘉女步行約 8-12 分鐘` |
| `milestones` | 關於頁的大事紀：每筆有 `year` / `title` / `desc`（可加 `link`）。`desc` 裡的 `{jobsAddress}` / `{shinobiAddress}` 會自動代入兩據點地址 | 2014 創立 → 2017 現址 → 2023 忍文理 → 2026 新官網 |

### 2.2 改法範例：換營業時間

假設改成「週一至週六 16:00–21:00」，`hours` 整段改成：

```json
"hours": {
  "openingHours": ["Mo-Sa 16:00-21:00"],
  "display": "週一至週六 16:00–21:00（週日休）",
  "opens": "16:00",
  "closes": "21:00",
  "days": ["Monday", "Tuesday", "Wednesday", "Thursday", "Friday", "Saturday"]
}
```

五個欄位**一起改、保持一致**（一個給 Google 簡寫、一個給人看、三個給結構化資料）。改完走檔案流程上線，Footer / 聯絡頁 / Google 結構化資料同時更新。**別忘了 §8：Google 商家檔案要另外手動改。**

### 2.3 社群連結的小例外

`info.json` 的 `socials` 是**預設值**；若 Vercel 環境變數（`PUBLIC_INSTAGRAM_URL` 等）有設定，**環境變數優先**。目前以 `info.json` 為準即可；若改了 `info.json` 但線上沒變，請工程師檢查 Vercel 環境變數是不是蓋住了它。

## 3. 價格與收退費：`src/content/fees/policy.json`

### 3.1 這個檔管什麼

| 區塊 | 內容 | 顯示在哪 |
|---|---|---|
| `coursePricing` | **每門課顯示的學費**：`default`（標準價）＋ `overrides`（特殊價課程逐一列出） | 課程內頁側欄、課程列表卡片、llms.txt |
| `methods` | 四種收費方式總覽（季繳 / 套裝 / 客製化 / 家教） | /fees |
| `quarterly` | 季繳細節：適用科目、價格表、季度劃分、早鳥與新生入班說明 | /fees |
| `tutoring` / `packageCourse` / `customized` | 家教計費、套裝、客製化的說明文字 | /fees |
| `payment` | 繳費方式（現金 / ATM）與「不另收教材費」註記 | /fees、課程頁側欄 |
| `refundGov` / `refundGovApplies` | 市府退費規定與適用對象 | /fees |
| `refundOldStudent` / `refundTable` | 季繳舊生優規退費資格與逐堂退費表 | /fees |

目前的 `coursePricing`：

```json
"coursePricing": {
  "default": "9,300 元 / 12 節（每節 3 小時）",
  "overrides": {
    "elementary-math": "9,300 元 / 24 節（每節 1.5 小時）",
    "python-programming": "11,400 元 / 12 節（每節 3 小時）"
  }
}
```

> 課程的 Markdown 檔（`src/content/courses/*.md`）**已經沒有價格欄位**——課程頁一律查這張表。所以「調價」永遠只改這一個檔，不會有哪一頁忘了改。

### 3.2 改價 SOP（標準情況）

1. 打開 `policy.json`
2. 改 `coursePricing`：
   - 全面調整標準價 → 改 `default` 一行
   - 某一門課特殊價 → 在 `overrides` 加一筆或改一筆，key 用課程檔名（如 `junior-math`）
3. **檢查 `quarterly.prices` 價格表是否要同步**（/fees 頁的表格與課程頁是兩個區塊，金額要一致）
4. 走檔案流程上線 → 課程頁、卡片、/fees、llms.txt 一次到位

### 3.3 程式設計課的「資料庫即時價」——實際行為（重要）

程式設計課另外接了 Supabase `course_prices` 表（key `programming_quarterly`）：資料庫有值時**優先於** `policy.json`。但經查證實際行為，請注意兩件事：

1. **改了 DB 也要 Redeploy 才會反映。** /fees 與課程頁都是部署時做好的靜態頁，資料庫是在「部署那一刻」被讀取的（程式碼註解裡寫的「不必 rebuild」與實際不符，以本手冊為準）。
2. **DB 只覆蓋兩個地方**：/fees 價格表的程式設計列、Python 課程內頁側欄。**課程列表卡片與 llms.txt 仍讀 `policy.json` 定值。**

> **因此建議的改價方式：直接改 `policy.json`（§3.2），一個動作全站一致。** 若您習慣在 Supabase 後臺改 `course_prices`，請務必「DB 與 policy.json 兩邊都改成同一個數字，然後 Redeploy」，否則卡片上會殘留舊價。

### 3.4 請假、補課等課務政策

書面政策條文放在 `src/content/policy/*.md`（學生請假、老師請假、颱風、補課券、家教補課、不可抗力，一個主題一個檔），顯示在 /policy 頁。改條文 = 改對應的 Markdown 檔，走檔案流程。

## 4. 課表與開課時段：Supabase 資料庫

### 4.1 一份資料、四個地方同時用

課表資料的唯一權威是 Supabase 的 **`teacher_availability` 表**（哪位老師、星期幾、在哪間教室）。系統把它組合成「開課清單」（view `class_offerings`），四個地方同源取用：

1. **/schedule 課表頁**
2. **試聽表單**的科目與時段選項（/contact 與每個課程頁）
3. **課程內頁的「上課時段」側欄**
4. **Google 結構化資料**（CourseInstance，讓 Google 知道何時開課）

好處：改一次排課，四處保證一致——不會發生「課表有開但試聽表選不到」這種尷尬。

### 4.2 業主改課表的 SOP

1. 登入 Supabase 後臺（dashboard）→ 專案 `jobs-official-site` → Table Editor
2. 改 **`teacher_availability`** 表：加列 = 某老師某天在某教室有課；刪列 = 取消
3. 相關參考表：`teachers`（老師名單與科目）、`courses`（課程清單）、`locations`（兩間教室）——一般排課只動 `teacher_availability`
4. **回到 Vercel 做一次 Redeploy**（§1.4）
5. 部署完成後，打開 /schedule 與 /contact 用手機確認

### 4.3 `src/content/schedule/*.json` 是什麼？——平常不要動它

repo 裡的 `src/content/schedule/`（barney.json、jason.json…）是**後備資料**：只有在網站建置時連不上資料庫的情況下才會被拿來用，讓網站永遠做得出來、不會開天窗。

> **平常不用維護它、也不要拿它來改課表**——連得上資料庫時它完全不生效，改了也看不到。真的要動它的時機只有一種：工程師調整後備資料時。

### 4.4 老師資料的「一名兩處」注意

- 老師的**介紹內容**（學歷、理念、照片）在 `src/content/teachers/<slug>.md`（檔案流程）。
- 課表與試聽表上顯示的**老師姓名與科目**來自 Supabase `teachers` 表。

所以：改介紹 → 只改 md；**改名字或科目 → md 與 DB 兩邊都改**（並 Redeploy），否則師資頁與課表會各講各話。

## 5. 新開一門課 / 下架一門課

這是全站目前**僅存**需要「一件事動兩處」的操作（課程頁內容在檔案、課表與試聽表吃資料庫），請照 checklist 逐項打勾。

### 5.1 新開課 checklist

- [ ] **1. 建課程頁**：新增 `src/content/courses/<slug>.md`（slug = 英數小寫加連字符的檔名，如 `high-biology`）。frontmatter 必填：`name`、`grade`（國小/國中/高中）、`subject`、`summary`；建議填：`teacher`（老師 slug）、`classType`、`trialLessons`、`lessonHours`、`cover`、`seoDescription`。內文寫課程介紹。**注意：不要寫價格、不要寫時段**（分別由 policy.json 與資料庫供應）。
- [ ] **2. 登記到資料庫**：Supabase `courses` 表加一列，**slug 必須與 md 檔名完全一致**，並填 name / grade / subject / teacher_slug / trial_lessons / display_order。
- [ ] **3. 排課**：Supabase `teacher_availability` 加上該老師的時段（若還沒排課，課程頁會顯示「實際開課時段請見課表頁」，不會壞，但建議盡快補）。
- [ ] **4. 若非標準價**：`policy.json` → `coursePricing.overrides` 加一筆 `"<slug>": "…元 / …節（每節…小時）"`。標準價 9,300 / 12 節則跳過。
- [ ] **5. 封面圖**：放進 `src/assets/images/courses/`，md 的 `cover` 指到它（規格見 §11）。
- [ ] **6. 上線**：檔案改動走檔案流程；DB 改動記得 Redeploy。最後用手機檢查：課程頁、/courses 列表、/schedule、試聽表單科目是否都出現了。

### 5.2 下架課 checklist

- [ ] 從 Supabase `courses` 表移除（或依工程師建議停用）該課程，`teacher_availability` 一併撤掉對應時段
- [ ] 刪除（或請工程師保留備份後移除）`src/content/courses/<slug>.md`
- [ ] 若 `coursePricing.overrides` 有這門課的特殊價，順手清掉
- [ ] 檢查 FAQ / 文章 / LP 內文有沒有提到這門課的字樣（快照內容不會自動跟，見 §7）
- [ ] 檔案流程上線 + Redeploy → 確認課程頁已消失、課表與試聽表不再出現該科目

## 6. SEO 自動連動：改資料 = 改 SEO

您**不需要**在改完資料後另外「處理 SEO」。以下四樣東西都在建置網站時自動從權威資料源取值重生：

| SEO 元件 | 白話說明 | 資料來源 |
|---|---|---|
| JSON-LD 結構化資料 | 藏在頁面裡給 Google 讀的「名片」：機構名稱、地址、電話、營業時間、課程、FAQ、開課時段 | info.json、policy.json、Supabase 課表、各內容檔 |
| OG 分享預覽圖 | 貼到 LINE / FB 時出現的預覽卡片 | 各頁標題與摘要（自動生成） |
| meta description | Google 搜尋結果裡那兩行描述 | info.json 的品牌簡介模板 + 各內容檔的 `seoDescription` / `summary` |
| llms.txt / llms-full.txt | 給 AI（ChatGPT 等）讀的全站摘要，讓 AI 回答關於本班的問題時引用正確事實 | 建置時自動重生，取自 info.json + policy.json + 全部內容檔 |

換句話說：**照 §0 的表把資料改在正確的地方，SEO 就已經改完了。** 唯一要人工做的是站外平台（§8）。

## 11. 圖片上傳建議

### 11.1 規格

| 用途 | 建議尺寸 | 最大檔案 | 格式 |
|---|---|---|---|
| 文章封面 | 1200 × 630 px | 500 KB | JPG / WebP |
| 師資大頭照 | 800 × 800 px（正方形） | 300 KB | JPG / WebP |
| 課程封面 | 1600 × 900 px | 500 KB | JPG / WebP |
| LP hero | 1920 × 1080 px | 800 KB | JPG / WebP |
| 文章內圖 | 寬 ≤ 1200 px | 400 KB | JPG / PNG / WebP |

### 11.2 命名與 alt

- 檔名**小寫英數加連字符**，不要中文與空格：好 `teacher-lin-portrait.jpg`；差 `林老師大頭.jpg`、`IMG 0123.jpg`

### 11.3 壓縮

1MB 以上的原始照片一定先壓縮（[TinyPNG](https://tinypng.com/)、[Squoosh](https://squoosh.app/)），否則手機開網站會慢。

## 12. 品牌守則速查

完整版見 [BRAND_GUIDELINES.md](../BRAND_GUIDELINES.md)。撰寫前花 30 秒掃過：

**多用**：適合 / 協助 / 陪伴 / 累積 / 踏實 / 穩 / 個別化 / 量身規劃 / 我們發現 / 在我們的經驗裡

> ✗ 業界第一、最強、最厲害

## 14. 常見問題

### Q1：我要調學費，改哪裡？

`policy.json` 的 `coursePricing`（§3.2）。**不要**去課程頁的檔案裡找價格欄位——已經沒有了，這是刻意的設計。

### Q2：我在 Supabase 改了課表，網站怎麼沒變？

正常。要多做一步 **Redeploy**（§1.4）。這是資料庫類改動唯一要記的事。

### Q3：我發現錯字怎麼改？

內容檔（文章 / FAQ / 師資 / 課程介紹）直接改對應 Markdown，走檔案流程。1-2 分鐘部署後生效。

### Q4：我不小心刪了內容 / 想看以前的版本？

所有檔案改動都在 Git 版本歷史裡，**先別緊張**，立刻聯絡工程師，告知大致時間與內容名稱，5-10 分鐘可救回。（資料庫的刪除沒有這層保護——刪 Supabase 資料列前請多看一眼。）

### Q5：我發了文章，但網站上看不到？

依序排查：① 還是草稿狀態？② 剛發佈不到 2 分鐘？③ 快取問題，用無痕視窗；④ 仍看不到 → 聯絡工程師。

### Q6：改了 `info.json` 之後 build 失敗？

這就是防呆在工作（§1.2）——通常是 JSON 格式錯了（少逗號、少引號、刪錯欄位）。把錯誤訊息貼給工程師，或對照 §2.1 的欄位表把格式改回來。改壞的版本**不會**上線，所以線上網站不受影響。

### Q7：明年要不要把「品牌 12 年」改成「13 年」？

不用。年資與版權年份由 `founded`（2014）自動推算，每年過年自動 +1。只有「最資深老師年資」（`stats.maxTeacherYears`）是手填的，想更新時改那一格。

### Q8：可以排程在未來某天才發佈嗎？

目前沒有自動排程。變通：內容先存草稿，到指定日再發佈；如需自動排程請工程師評估（屬額外功能）。

### Q9：標籤（tags）建議怎麼用？

一篇 3-5 個、用詞統一（都用「學測」就別再寫「學科能力測驗」）。常用：學測 / 會考 / 數學 / 物理 / 化學 / 英文 / 高三 / 國三 / 考前準備 / 學習方法。

### Q10：多人可以同時編輯嗎？

技術上可以，但建議錯開：同一份內容兩人同時改，後存的會蓋掉先存的。改之前在 LINE 群組吭一聲。

## 15. 求助路徑

### 何時自己處理 vs 何時找工程師

| 狀況 | 建議處理方式 |
|---|---|
| 改錯字、新增文章 / FAQ / 見證、改課程介紹 | 自己改（或請工程師代改） |

| 改課表、排課 | 自己在 Supabase 改 + Redeploy（§4） |

| 網站整個打不開 | 立刻找工程師 |

## 16. 著陸頁（LP）快速複製 Playbook

> 對應 RFP §1.2.4「LP 快速複製機制」。LP 是針對單一行銷活動的獨立專頁（寒暑假班、學測衝刺、廣告到達頁）。版型固定（hero 大標＋4 格賣點＋內文＋倒數計時＋試聽表單），只要換文字；路由 `/lp/[campaign]`，**刻意不進 sitemap、不被搜尋引擎索引**（流量靠廣告 / LINE / QR code 主動分享）。活動結束把 `published` 關掉即下架，檔案保留供下次複製。

### 16.1 複製檔案

```bash
# 在專案根目錄執行；範本 summer-2026.md 內文已寫滿課程、學費、試聽區塊
cp src/content/landing/summer-2026.md src/content/landing/winter-2026.md
```

檔名規則：小寫英數加連字符。路由由 frontmatter 的 `campaign` 決定，建議檔名與 `campaign` 一致。

### 16.2 編輯 frontmatter

欄位定義在 `src/content/config.ts` 的 `landing` collection：

| 欄位 | 必填 | 說明 |
|---|---|---|
| `campaign` | ✅ | URL 路由名 → `/lp/<campaign>`，全站唯一 |
| `title` | ✅ | 瀏覽器分頁標題 / SEO 標題 |
| `headline` | ✅ | hero 大標（H1） |
| `subheadline` | 選 | 大標下方說明 |
| `heroImage` | 選 | hero 主視覺（相對路徑；暫不需要可整行刪除） |
| `ctaLabel` / `ctaHref` | 選 | 按鈕文字（預設「立即試聽」）/ 連結（預設 `/contact`） |
| `startDate` / `endDate` | 選 | 活動起訖；未過期會顯示倒數計時 |
| `published` | 選 | `true` 生成頁面；`false` 下架（預設 `true`） |

範例：

```yaml
---
campaign: winter-2026
title: 2026 寒假班｜開學前最後衝刺
headline: 寒假，是補回上學期落差的黃金 30 天
subheadline: 1/20 - 2/15，數理英文小班，從盲點診斷到題型補強一次到位。
ctaLabel: 立即預約試聽
ctaHref: /contact
startDate: 2026-01-20
endDate: 2026-02-15
published: true
---
```

`---` 以下的內文改成新活動內容（Markdown 見 §10，文案對齊 §12）。

### 16.3 本地預覽

`npm run dev` → 開 `http://localhost:4321/lp/winter-2026`。檢查：大標 / 日期 / 4 格賣點 / 倒數計時 / 底部試聽表單 / 手機寬度（390px）。新增檔案後若 404，重啟 dev server 讓內容重新掃描。

### 16.4 上線注意事項

- **自動發現**：檔案放進 `src/content/landing/` 且 `published: true`，build 時自動產生頁面，不需登記清單；`campaign` 重複會 build 報錯。
- **不進 sitemap 是刻意的**（`astro.config.mjs` 的 sitemap filter 排除 `/lp/`）：LP 是短期到達頁，不希望與正式課程頁在自然搜尋互相競爭。若真要讓某 LP 被索引（少見），屬工程決策，開 issue 討論、不要逕自改設定。
- **部署**：照 §1.3 檔案流程；preview 確認 → 業主拍板上線。

最短路徑：複製檔 → 改 `campaign` / `title` / `headline` ＋內文 → 本地預覽 → 檔案流程上線。全程不碰任何程式與設定檔。
